# KernelOn · 产品规划说明书

## 一、背景与目的

### 1.1 背景

新员工的入职、培养、导师匹配、过程跟踪等运作环节，目前缺乏统一的承载工具，信息分散、流程零散、体验割裂。作为部门新员工负责人，需要一个**集中、可扩展、且体验出色**的平台，把分散的管理动作整合到一处。

### 1.2 目的

本平台希望同时达成三个层面的目标：

| 层面         | 目标                                                                                                                         |
| ------------ | ---------------------------------------------------------------------------------------------------------------------------- |
| **管理目标** | 让新员工运作（入职、导师、培训、考核、跟踪）有统一抓手，可沉淀、可复用、可度量。                                             |
| **体验目标** | 新员工与管理者使用顺畅高效，打破传统后台管理系统的沉闷，体验丝滑，产品视觉表现令人叹为观止。                                 |
| **价值传递** | 通过极具科技感的 Web OS 架构，全方位展示部门在新员工管理上的精细化投入，以及在 AI 应用、Vibe Coding 与前端设计上的前沿造诣。 |

## 二、产品理念

### 2.1 核心隐喻：把“管理平台”做成一个“工作台 OS”

借鉴成熟桌面操作系统（如 macOS / iOS）的交互范式与美学质感，建立**自有的高拟真设计语言**（主题设计、图标体系、命名与动效），做到“神似其体验、自成其风格”，而非像素级复刻。

### 2.2 设计主张

1. **功能即应用（App 化）**：每一类管理能力都内聚为一个独立 App，职责清晰、可独立演进、可按需增减。
2. **开箱即用（预置清单）**：平台默认内置新员工必备 App 清单与常用小组件，新人登录即上手，无需配置。
3. **炫酷与效率一体**：精致高级的视觉表现贯穿平台始终，是体验的底色而非某一局部的装饰；与此同时，高频操作通过小组件与快捷入口实现一步直达。好看与好用相互成就，而非彼此妥协。

## 三、品牌设计：KernelOn

本平台正式命名为 **KernelOn**。这个名字不仅承载了极客的技术审美，更将产品形态与团队管理逻辑进行了深度融合。

命名寓意：

- **技术层（Kernel On / 内核启动）**：`Kernel` 在计算机科学中是系统的底层内核。本平台作为类 OS 架构，`KernelOn` 代表着一套高内聚、微应用架构的底层操作系统内核已经正式“启动与激活（On）”。它向所有人宣告了我们在前沿 Web 架构、AI 协同开发（Vibe Coding）上的硬核追求。
- **人才层（激活核心潜能）**：新员工和骨干导师是部门未来发展的“核心内核（Kernel）”。`KernelOn` 寓意着新员工通过本平台能够快速融入，激活个人的职业潜能，让团队的“核心引擎”轰鸣运转。

## 四、产品实现规划

### 4.1 整体形态

平台采用 **“外壳（Shell）+ 核心服务（Core）+ 应用（Apps）”** 的解耦架构。

```
┌────────────────────────────────────────────────────────┐
│                      DESKTOP SHELL                     │
│   (MenuBar, Dock, Grid Wallpaper, Multi-Screen Engine) │
└──────────────────────────┬─────────────────────────────┘
                           ▼
┌────────────────────────────────────────────────────────┐
│                    OS CORE SERVICES                    │
│   (Window Manager, Drag & Drop, State DB, AI Engine)   │
└──────────────────────────┬─────────────────────────────┘
                           ▼
┌────────────────────────────────────────────────────────┐
│                         APPS                           │
│   ┌───────────────┐ ┌───────────────┐ ┌──────────────┐ │
│   │  Onboarding   │ │  Mentor App   │ │ Widget Center│ │
│   └───────────────┘ └───────────────┘ └──────────────┘ │
└────────────────────────────────────────────────────────┘
```

在视觉表现上，平台呈现为一个类 MacOS/IOS 的统一工作台，用户登录后进入“桌面”，通过 App、小组件、Dock 等完成全部新员工运作。

基础能力设计：

- 桌面是一块**自由磁吸式画布**：App 图标与小组件可由用户随意摆放、混合排布于同一页面。
- **磁吸式桌面栅格系统 (Magnetic Grid)**：图标（1x1 栅格）和小组件（2x2 或 2x4 栅格等）在拖拽时会自动排开、对齐与吸附，既保留了 iOS 般丝滑的自由交互，又在系统层面保证了界面的整洁美观，降低布局混乱度。
- **多屏切换**：支持创建多个桌面屏，通过鼠标拖拽 / 滑动手势在屏间流畅切换。
- **常驻 Dock**：底部 Dock 固定常用 App，跨屏始终可见。
- **顶部状态栏**：承载身份、通知与全局搜索等系统级入口。
- **桌面与窗口**：App 以窗口形式打开，支持多窗口、拖拽、层级管理。
- **Dock / 启动台**：常用 App 常驻，全部 App 可在启动台总览与检索。
- **小组件（Widget）**：在桌面直接展示关键信息与快捷操作（待办、进度、快速录入等）。
- **按需加载**：App/Widget 的代码与资产可随平台或客户端预置在构建产物中，但只有被用户添加到桌面或主动打开时才加载、执行并渲染。
- **个性化**：支持更换壁纸、自由布局小组件与图标、自定义多屏桌面。
- **AI Spotlight**：按下 `Cmd + Space` 快速唤起。不仅是全局检索，还集成了 AI 问答、AI 指令（输入“匹配导师张三和新人李四”直接调起匹配界面）以及强大的AI Agent能力。
- **通知中心**：统一汇聚待办、提醒、流程通知。支持与钉钉/企业微信/飞书的 Webhook 联动。

### 4.2 业务 App 规划

## 五、平台支持范围与技术选型

- **Web 端**：首要交付目标，也是产品体验、业务流程、权限模型、数据模型与 AI 能力的主验证面。
- **桌面端**：基于 `Tauri + Vite` 独立建设桌面壳，复用 `packages/core`、`packages/catalog`、`packages/modules`、`packages/ui` 与 `packages/shell`，不强行把 Next.js server runtime 塞进桌面端。
- **移动端**：不在本期范围内。后续如需移动适配，应优先判断是响应式查看、轻量审批，还是完整移动产品。

### 5.1 最终主技术栈

为了以“一次性把事情做对”为目标，KernelOn 不把首期工程视为一个纯前端演示项目，而是按长期平台建设来设计技术栈。

| 层级           | 选型                                                      | 定位                                                                                                                                          |
| -------------- | --------------------------------------------------------- | --------------------------------------------------------------------------------------------------------------------------------------------- |
| Web 应用框架   | `Next.js App Router`                                      | 主 Web 应用框架，承载路由、布局、Server Components、Server Actions、Route Handlers 与后续权限边界。                                           |
| UI 运行时      | `React 19`                                                | Web OS Shell 与业务 App 的统一组件模型。交互壳层使用 Client Components，数据读取和页面装配默认靠 Server Components。                          |
| 类型系统       | `TypeScript`                                              | 作为核心领域模型、前后端契约、App manifest、窗口模型、命令模型的统一静态约束。                                                                |
| 样式系统       | `Tailwind CSS 4`                                          | 通过 `@theme` 统一设计 token，避免在 JavaScript 中重复维护主题常量。                                                                          |
| UI primitives  | `shadcn/ui` 思路 + `Radix Primitives`                     | Dialog、Popover、Slot、可访问性交互组件优先基于成熟 primitives 封装，不重复造基础交互轮子。                                                   |
| 动效           | `Motion`                                                  | 默认动效方案，用于窗口、启动台、Dock、Spotlight 等与 Web OS 隐喻强相关的交互。`GSAP`、`Three.js` 仅在高级时间线动画或 3D 场景明确需要时引入。 |
| 客户端 UI 状态 | `Zustand`                                                 | 只用于窗口栈、Dock、启动台、Spotlight、桌面屏幕、小组件布局等本地 Shell 状态。                                                                |
| 服务端数据     | `Server Components` / `Server Actions` / `Route Handlers` | 读写边界优先放在服务端，后续复杂客户端缓存或乐观更新再引入 TanStack Query。                                                                   |
| 数据层         | `PostgreSQL + Drizzle ORM`                                | 后续承载新员工、导师、成长档案、任务、考核、通知、资源等结构化数据。                                                                          |
| AI 能力        | 服务端 AI SDK / provider adapter                          | AI Spotlight、智能匹配、总结与 Agent 能力都走服务端适配层，不在客户端暴露模型密钥。                                                           |
| 包管理         | `pnpm workspace`                                          | 支撑 monorepo 下 Web、未来桌面端、核心模型、UI 系统、Shell 与后续数据/AI 包的清晰边界。                                                       |

### 5.2 状态与边界原则

- `packages/core` 是纯 TypeScript 内核，不依赖 React，承载 App manifest、窗口模型、桌面布局、命令模型等可测试的纯能力。
- `packages/catalog` 是 App/Widget 注册目录与默认桌面配置层。它只保存 manifest、分类、默认布局、`loaderKey` 等可序列化描述，不直接导入业务 React 组件。
- `packages/modules` 是可动态加载的业务模块层。App 窗口和 Widget 通过运行时注册表按 `loaderKey` 懒加载，支持未来按组织、角色、用户个性化配置裁剪加载范围。
- `packages/ui` 是设计系统原语层，沉淀 Button、IconButton、Surface、WindowFrame 等可复用组件，并逐步接入 shadcn/Radix 风格的可访问性基础组件。
- `packages/shell` 是客户端 Web OS 壳层，承载桌面、Dock、启动台、窗口层、Spotlight 与本地 UI store。Shell 不直接绑定具体业务 App，而是消费 manifest、桌面布局和运行时注册表。
- `apps/web` 是 Next.js 应用装配层，负责路由、布局、服务端数据入口、权限边界和业务 App 组合。
- `apps/desktop` 是 Tauri + Vite 桌面壳层，负责窗口容器、桌面构建、原生能力桥接和共享模块装配；除原生集成外，不重复实现 Web 已有业务组件。
- Zustand 不管理远端列表缓存、复杂表单工作流、审批流状态机或跨用户协同状态；这些能力后续分别由服务端模型、数据库事务、工作流建模或专用协同方案承载。

### 5.3 桌面端策略

Tauri 是桌面端的正确方向，但不是首期 Web 主应用的约束。原因是 Next.js 的长期价值在服务端边界、权限、数据读取与平台能力，而 Tauri 更适合作为后置桌面壳。

因此最终策略是：

- 主 Web：`apps/web` 使用 Next.js App Router。
- 桌面壳：`apps/desktop` 使用 `Tauri + Vite`，并复用共享包。
- 共享能力：核心模型、App/Widget 注册目录、可动态加载模块、UI 原语、Shell 交互尽量沉淀在 `packages`，避免 Web 与桌面各写一套。

这套策略既保留 Next.js 对平台型 Web 应用的长期优势，也为桌面端留下轻量、高性能、可控的实现路径。
